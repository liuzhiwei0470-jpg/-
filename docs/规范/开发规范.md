# 开发规范

## 一、代码规范

### 1.1 TypeScript规范

#### 命名规范
| 类型 | 规范 | 示例 |
|------|------|------|
| 变量 | camelCase | `userName`, `isValid` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 函数 | camelCase + 动词 | `getUserData`, `fetchArticles` |
| 类名 | PascalCase | `UserService`, `ArticleModel` |
| 接口 | PascalCase | `IUser`, `IArticle` |
| 类型 | PascalCase | `UserType`, `ArticleStatus` |

#### 文件命名
| 类型 | 规范 | 示例 |
|------|------|------|
| 工具类 | kebab-case | `date-utils.ts`, `format-helper.ts` |
| 组件 | PascalCase | `ArticleList.vue`, `UserCard.vue` |
| 配置文件 | camelCase/kebab-case | `vite.config.ts`, `.env.example` |
| 类型定义 | PascalCase | `types.ts`, `user.types.ts` |

#### 代码格式
- 使用2空格缩进
- 语句末尾加分号
- 对象最后一项不加逗号
- 优先使用 `const`，其次 `let`，不用 `var`

### 1.2 Vue规范

#### 组件结构
```vue
<template>
  <!-- 模板 -->
</template>

<script setup lang="ts">
// 逻辑
</script>

<style scoped>
/* 样式 */
</style>
```

#### 组件命名
- 基础组件：`Base` 前缀，如 `BaseButton.vue`
- 业务组件：特性命名，如 `ArticleList.vue`
- 组件文件：PascalCase

#### API调用
- 统一放在 `src/api/` 目录
- 一个模块一个文件，如 `user.ts`, `article.ts`
- 使用 `async/await`
- 统一错误处理

### 1.3 Node.js/Express规范

#### 目录结构
```
backend/
├── routes/          # 路由
├── controllers/     # 控制器
├── services/        # 业务逻辑
├── models/          # 数据模型
├── middleware/      # 中间件
├── utils/           # 工具函数
└── config/          # 配置文件
```

#### 错误处理
```typescript
// 统一错误格式
{
  success: false,
  error: {
    code: 'ERROR_CODE',
    message: '错误描述'
  }
}
```

## 二、Git规范

### 2.1 分支命名

| 分支类型 | 命名规则 | 示例 |
|----------|----------|------|
| 主分支 | main | main |
| 开发分支 | develop | develop |
| 功能分支 | feature/功能名 | feature/user-auth |
| 修复分支 | fix/问题描述 | fix/login-bug |
| 发布分支 | release/版本号 | release/v1.0.0 |

### 2.2 提交信息格式

```
<类型>: <简短描述>

[可选的详细描述]
```

#### 类型
| 类型 | 说明 |
|------|------|
| feat | 新功能 |
| fix | Bug修复 |
| docs | 文档更新 |
| style | 代码格式（不影响功能） |
| refactor | 重构 |
| test | 测试相关 |
| chore | 构建/工具相关 |

#### 示例
```
feat: 添加用户注册功能

- 支持邮箱注册
- 添加密码强度验证
- 发送注册成功邮件
```

### 2.3 Commit Message Rules

- 标题不超过50字
- 描述不超过72字
- 使用中文描述

## 三、文档规范

### 3.1 README.md

每个项目必须包含：
- 项目简介
- 功能列表
- 技术栈
- 快速开始
- 环境变量说明

### 3.2 代码注释

- 复杂逻辑必须注释
- 保持注释与代码同步
- 使用中文注释

### 3.3 API文档

使用以下格式：

```markdown
### GET /api/users

获取用户列表

**请求参数：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | number | 否 | 页码 |

**响应示例：**
```json
{
  "success": true,
  "data": []
}
```
```

## 四、数据库规范

### 4.1 表命名
- 使用复数名词：`users`, `articles`, `subscriptions`
- 小写下划线分隔

### 4.2 字段命名
- 小写下划线分隔：`user_id`, `created_at`
- 主键：`id`
- 外键：`{表名}_id`
- 时间字段：`created_at`, `updated_at`
- 布尔字段：`is_{状态}`, `has_{属性}`

### 4.3 SQL格式化
- 关键字大写
- 表名和字段名加反引号
- 使用参数化查询

---

## 五、安全规范

- 敏感信息（密码、Token）严禁硬编码
- 使用环境变量存储敏感配置
- 用户输入必须验证和过滤
- SQL注入防护（参数化查询）
- XSS防护（转义输出）
- CSRF防护

---

## 六、测试规范

### 6.1 单元测试
- 核心业务逻辑必须有单元测试
- 使用 Vitest（前端）/ Jest（后端）

### 6.2 测试覆盖率
- 核心模块覆盖率 > 80%
